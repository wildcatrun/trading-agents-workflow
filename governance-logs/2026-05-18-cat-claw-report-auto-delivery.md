# Cat Claw Report Auto Delivery

Timestamp: 2026-05-18T12:25:00+08:00

## Problem

Cat Claw could produce a formal `workflow_secretary_report`, but the report
could remain in `telegram_outbox` with status `queued`. Flashcat did not receive
the message until Codex manually ran `telegram-outbox --deliver`.

This created a false completion state:

- runtime dispatch was `acked`
- report text was persisted
- outbox row existed
- Telegram delivery receipt was missing

## Root Cause

`workflow.supervise` created the Cat Claw report dispatch but did not drain that
report dispatch inside the same supervisor turn. When the report was drained
manually, `meeting.ingest` created a private Telegram outbox item, but
`runtime-bridge` did not deliver report outbox rows automatically.

## Plugin Fix

The fix stays inside `trading-agents-workflow`:

1. `runtime-bridge` now auto-delivers report outbox rows generated by
   `workflow_secretary_report` and `human_gate_report`.
2. `workflow.supervise --drain` now drains the Cat Claw report dispatch it just
   created, using the exact dispatch id instead of draining a generic runtime
   queue item.
3. The runtime bridge result now exposes `reportDelivery`, including `sent` or
   `failed` status and Telegram receipts.

No new external script, timer, relay, or parallel message system is required.

## Expected Behavior

When `workflow.supervise --drain` reaches a state that requires Cat Claw closeout:

1. Create the `workflow_secretary_report` or `human_gate_report` dispatch.
2. Drain that exact dispatch.
3. Ingest Cat Claw output.
4. Create private Telegram outbox for Flashcat if needed.
5. Deliver it immediately through the governed OpenClaw Telegram path.
6. Store Telegram receipts in `telegram_outbox.payload_json.delivery`.
